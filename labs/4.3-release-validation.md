# Lab 4.3 – Release Validation and Monitoring

**Duration:** ~60 minutes  
**Format:** Hands-on — individual or pairs

---

## Learning Objectives

By the end of this lab you will be able to:

- Promote a release to production using the deployment pipeline
- Validate a production deployment using the regression test suite
- Enable and navigate ECS Container Insights in CloudWatch
- Build a CloudWatch dashboard to monitor a live production service
- Create alarms with meaningful thresholds tied to release health criteria
- Use CloudWatch Logs Insights to detect error patterns in application logs
- Apply post-release metrics to make a data-driven go/no-go decision

---

## Background

In previous labs you promoted `v2.0.0` through dev and QA and updated your regression test suite to cover the v2 API contract. Version `v2.0.1` is now sitting in QA — a patch release with bug fixes and no breaking API changes. The v2 test suite will cover it without modification.

In this lab you will promote `v2.0.1` to production, run the regression suite as a release gate, and then build the monitoring layer that tells you whether the deployment is actually healthy: Container Insights, a CloudWatch dashboard, alarms, Logs Insights queries, and a curl health probe. At the end, your instructor will inject a fault and you will use what you built to make a go/no-go call.

## Part 1 — Deploy v2.0.1 to Production (10 minutes)

### Run the deployment pipeline

1. In Bitbucket, navigate to **Pipelines → Run pipeline**.
2. Select the `deploy-prod` custom pipeline.
3. Set `TAG` to `v2.0.1` and click **Run**.
4. Wait for the pipeline to complete. A green checkmark means the ECS service has been updated and the new task is running.

> ECS will start a new task with the updated image and stop the old one. The service's public IP will change — you will need the new IP in Part 5.

### Confirm the deployment

1. Open the ECS Console → **Clusters** → **order-match-api** → **Services** → **order-match-api-prod**.
2. On the **Tasks** tab, confirm one task is in **RUNNING** status.
3. Note the new **Public IP** — you will use it later.

### Run regression tests

With the deployment confirmed, run the regression suite as the release gate.

1. In Bitbucket, run the `regression-tests` custom pipeline with these variables:
   - `BASE_URL` = `http://<prod-ip>:8000` (the public IP from above)
   - `STAGE` = `prod`
   - `VERSION` = `v2`
2. Wait for the pipeline to complete and open the **Tests** tab.
3. All tests must pass before continuing. If any test fails, stop — do not proceed with the monitoring setup until you understand why.

## Part 2 — Enable Container Insights (5 minutes)


By default ECS publishes basic CPU and memory metrics to CloudWatch, but Container Insights unlocks richer task-level data. Enable it on the cluster now.

1. Open the [AWS ECS Console](https://console.aws.amazon.com/ecs).
2. Click **Clusters** in the left nav and select **order-match-api**.
3. Click **Update Cluster** (top right).
4. Under **Monitoring**, set **Container Insights** to **Enhanced**.
5. Click **Update**.

> Container Insights takes 1–2 minutes to start populating metrics. Continue to Part 3 while it warms up.

---

## Part 3 — Build a CloudWatch Dashboard (15 minutes)

You will build a dashboard with four widgets that give a complete picture of production service health.

### Create the dashboard

1. Open the [CloudWatch Console](https://console.aws.amazon.com/cloudwatch).
2. Click **Dashboards** → **Create dashboard**.
3. Name it `order-match-api-prod` and click **Create dashboard**.
4. When prompted to add a widget, click **Cancel** — you will add them manually below.

---

### Widget 1 — CPU Utilization

1. Click **Add widget** → **Line** → **Next**.
2. Select **Metrics** → **Browse**.
3. In the search bar type `ECS` and select **ECS → ClusterName, ServiceName**.
4. Find `order-match-api` / `order-match-api-prod` → check **CPUUtilization**.
5. Click **Create widget**.

---

### Widget 2 — Memory Utilization

Repeat the same steps as Widget 1, but select **MemoryUtilization** instead of CPUUtilization.

---

### Widget 3 — Running Task Count

1. Click **Add widget** → **Number** → **Next**.
2. Select **Metrics** → **Browse** → **ECS → ClusterName, ServiceName**.
3. Find `order-match-api` / `order-match-api-prod` → check **RunningTaskCount**.
4. Click **Create widget**.

> This is your most important signal. If RunningTaskCount drops to 0 the service is down.

---

### Widget 4 — Log Event Volume

1. Click **Add widget** → **Line** → **Next**.
2. Select **Logs** tab → **Browse log groups**.
3. Select `/ecs/order-match-api/prod`.
4. In the query box enter:

```
fields @timestamp, @message
| stats count() as eventCount by bin(1m)
```

5. Set the time range to **Last 30 minutes**.
6. Click **Create widget**.

Save the dashboard.

## Part 4 — Create CloudWatch Alarms (10 minutes)

Alarms turn metrics into actionable signals. Create two alarms for the production service.

### Alarm 1 — High CPU

1. In CloudWatch, click **Alarms** → **All alarms** → **Create alarm**.
2. Click **Select metric** → **ECS → ClusterName, ServiceName**.
3. Select `order-match-api` / `order-match-api-prod` → **CPUUtilization** → **Select metric**.
4. Configure the conditions:
   - **Threshold type:** Static
   - **Condition:** Greater than **80**
   - **Datapoints to alarm:** 2 out of 2
   - **Period:** 1 minute
5. Skip notification actions for now → click **Next**.
6. Name the alarm `prod-cpu-high` → click **Create alarm**.

---

### Alarm 2 — Task Count Below Minimum

1. Create another alarm → **ECS → ClusterName, ServiceName**.
2. Select `order-match-api` / `order-match-api-prod` → **RunningTaskCount** → **Select metric**.
3. Configure the conditions:
   - **Threshold type:** Static
   - **Condition:** Lower than **1**
   - **Datapoints to alarm:** 1 out of 1
   - **Period:** 1 minute
4. Name the alarm `prod-task-down` → **Create alarm**.

## Part 5 — Logs Insights Queries (10 minutes)

CloudWatch Logs Insights lets you query application logs directly. Run each of the following queries against the production log group and record what you find.

1. In CloudWatch, click **Logs Insights**.
2. In the **Select log group(s)** dropdown, select `/ecs/order-match-api/prod`.
3. Set the time range to **Last 1 hour**.

---

### Query 1 — Log Volume Over Time

```
fields @timestamp, @message
| stats count() as requests by bin(5m)
| sort @timestamp asc
```

**What to look for:** Is traffic fairly steady? Are there gaps that might indicate the service was down?

---

### Query 2 — Error Scan

```
fields @timestamp, @message
| filter @message like /(?i)error/
| sort @timestamp desc
| limit 20
```

**What to look for:** Any error messages? What are they? Are they recurring?

---

### Query 3 — HTTP 5xx Responses

```
fields @timestamp, @message
| filter @message like /5[0-9][0-9]/
| sort @timestamp desc
| limit 20
```

**What to look for:** Are there 500-level responses in the logs? How recent are they?

> If the log group has no data yet — the service may not have received traffic. Run `seed-prod` from Bitbucket Pipelines to generate some, then re-run the queries.

## Part 6 — Synthetic Health Probe (5 minutes)

A synthetic probe simulates a real user request to verify the API is alive and responding correctly. You will hit the production task directly.

### Get the task's public IP

1. In the ECS Console, click **Clusters** → **order-match-api**.
2. Click the **Tasks** tab → click the running `order-match-api-prod` task.
3. Under **Configuration**, copy the **Public IP**.

### Run the probe

Replace `<IP>` with the public IP you copied.

```bash
curl -s -o /dev/null -w "%{http_code} %{time_total}s\n" http://<IP>:8000/orders
curl -s -o /dev/null -w "%{http_code} %{time_total}s\n" http://<IP>:8000/queue
```

Record your results:

| Endpoint | Status Code | Response Time | Healthy? |
|----------|------------|---------------|---------|
| `GET /orders` | | | |
| `GET /queue` | | | |

A `200` with a response time under 1 second is a passing probe. Anything else warrants investigation.

Your instructor will now trigger a fault against the production service. Watch your dashboard.

**What the instructor does:** Runs the `reset-prod` Bitbucket pipeline, which restarts the ECS service and briefly brings the running task count to 0.

**What you should observe:**
- `RunningTaskCount` drops — does your `prod-task-down` alarm trigger?
- `CPUUtilization` spikes briefly on restart
- Log volume pauses then resumes

## Summary

In this lab you:

- Enabled Container Insights on the ECS cluster to unlock task-level metrics
- Built a CloudWatch dashboard covering CPU, memory, running task count, and log volume
- Created alarms with meaningful thresholds — a high-CPU warning and a task-down critical
- Used Logs Insights to scan production logs for errors and traffic patterns
- Ran a synthetic probe against the live task to verify endpoint availability and response time
- Applied all of the above to make a data-driven go/no-go call after a fault was injected

This is the monitoring baseline for any production service. The signals you set up today — task health, error rates, response times, log anomalies — are the same signals that page an on-call engineer at 2am and give a release manager confidence to stand down.
