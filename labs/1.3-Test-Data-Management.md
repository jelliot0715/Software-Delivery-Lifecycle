---

## Test Data Management

Test environments need data that realistically represents production to produce valid results. An empty dev environment gives false confidence — a service can pass every test simply because there is nothing to process. At the same time, the data cannot come from production. TDM (Test Data Management) is the practice of ensuring each environment has appropriate, controlled, realistic data that reflects what production looks like, without using production data to do it.

---

## Why Production Data Cannot Be Used in Test Environments

Production data contains PII — names, email addresses, financial details, customer IDs. Regulations like GDPR and HIPAA prohibit using real customer data outside of production systems. A data breach in a dev or QA environment carries the same legal consequences as one in production. Even internal access by developers to real customer records can create compliance violations. The simplest way to avoid all of this is to never move production data out of production in the first place.

---

## Synthetic Data

Synthetic data is generated. It has the same structure and statistical distribution as production data but contains no real information. It can be regenerated at any time, which makes environment refresh cycles straightforward and consistent. For the Order Match API, synthetic data means a realistic spread of buy and sell orders at market-like prices and quantities — data that exercises the same code paths as a live system without involving any real users.

---

## Environment Refresh Cycles

Test environments accumulate stale data over time — cancelled orders, test artifacts, records created during previous test runs. Left unmanaged, this creates noise that makes test results harder to interpret. Teams define a refresh cycle, typically weekly, where the environment is reseeded with a fresh synthetic dataset. A pipeline step makes the refresh repeatable and produces an identical starting state across all environments.

## Hands On

In this exercise you will write a data seeding script for the Order Match API, run it against your dev environment, and add a seed pipeline step so the refresh can be triggered from Bitbucket on demand.

1. **Write the seeding script.**

   Create a `tests/` directory in the root of your forked repo and add `tests/seed.py`:

   ```python
   import os
   import random
   import uuid
   import requests

   BASE_URL = os.environ.get("BASE_URL", "http://localhost:8000")
   NUM_ORDERS = 20


   def seed():
       sides = ["buy", "sell"]
       for _ in range(NUM_ORDERS):
           body = {
               "id": str(uuid.uuid4()),
               "side": random.choice(sides),
               "price": random.randint(50, 200),
               "quantity": random.randint(1, 100),
               "status": "pending",
           }
           r = requests.post(f"{BASE_URL}/orders", json=body)
           r.raise_for_status()
       print(f"Seeded {NUM_ORDERS} orders to {BASE_URL}")


   if __name__ == "__main__":
       seed()
   ```

   `uuid.uuid4()` generates a unique ID for each order. `random.randint(50, 200)` produces integer prices across a realistic range. `random.randint(1, 100)` varies order size. `r.raise_for_status()` surfaces API errors immediately, so the script stops on the first failed request rather than silently continuing.

2. **Run the script locally (optional).**

   ```
   pip install requests
   BASE_URL=http://<dev-ip>:8000 python tests/seed.py
   ```

   Then verify the data landed:

   ```
   curl http://<dev-ip>:8000/orders
   ```

   The response should contain 20 orders at varied prices, quantities, and sides. This is a synthetic dataset — no real customer data was used to generate it.

3. **Add the seed pipeline step.**

   Open `bitbucket-pipelines.yml` and add a `seed-dev` custom pipeline:

   ```yaml
   seed-dev:
     - variables:
         - name: BASE_URL
           description: "Base URL of the dev environment (e.g. http://<public-ip>:8000)"
     - step:
         name: Seed Test Data – Dev
         image: python:3.11-slim
         script:
           - pip install requests -q
           - python tests/seed.py
   ```

   Only the `requests` library is needed here — this is data setup, not test assertion. The script exits non-zero if any POST request fails, which causes the pipeline step to fail and makes the problem visible immediately.

4. **Commit and push.**

   Stage `tests/seed.py` and the updated `bitbucket-pipelines.yml`, then push to your repo.

5. **Run the seed pipeline.**

   In Bitbucket, navigate to **Pipelines → Run pipeline**. Select the `seed-dev` custom pipeline. Enter the dev service's public IP as `BASE_URL` (e.g., `http://1.2.3.4:8000`). Run it. Confirm the pipeline output shows "Seeded 20 orders to http://...".

6. **Verify the data.**

   Navigate to `http://<dev-ip>:8000/orders` in a browser. Confirm the environment contains 20 synthetic orders with varied prices and quantities. Run the seed pipeline a second time. Confirm a second batch of 20 orders is added. Each run is independent and additive; the environment does not reset between seeds unless the service itself is restarted.

7. **Challenge: add a refresh cycle for all three environments.**

   Add `seed-qa` and `seed-prod` custom pipeline steps to `bitbucket-pipelines.yml` following the same pattern as `seed-dev`. Run all three pipelines. This represents a weekly refresh — each test cycle begins with a consistent, controlled dataset across every environment.
