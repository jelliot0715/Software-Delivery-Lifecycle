## Service Delivery & SDLC Overview

**Software Development Lifecycle (SDLC)** describes how software moves from initial requirements through development, testing, deployment, and ongoing production support.

**ITIL v4** (Information Technology Infrastructure Library) is a framework that provides a common language and best practices for managing IT services across their lifecycle.

IT systems are treated as **services**. All services managed by a team are organized within a **service portfolio**. The goal of every service is to deliver **value**.

In the early stages of the SDLC, Business Analysts (BA) identify new services or enhancements that will provide value to customers. 

BAs and Project Managers (PM) plan the delivery or enhancement of a service, considering risk, cost, ROI, and long-term support.

Product Owners (PO) manage the **service backlog** and priorities, while development and operations teams handle implementation and ongoing support. They may use SCRUM or some other project management technique.

Each role has a distinct focus within the SDLC:

| Role | Focus |
|------|-------|
| PO | “What should we build next?” |
| BA | “What problem are we solving?” |
| PM | “Are we delivering on time / within scope?” |
| Devs | “Build it.” |
| Ops/SRE | “Keep it running.” |

## Staging

To develop, test, and deploy software to production, we use separate environments called stages.
Each stage is designed to closely resemble production, while often using fewer resources to reduce cost.

Stages act as a controlled testing ground where services are validated and prepared for release.

A common staging strategy uses three environments: **dev**, **QA**, and **prod**. Different companies may add more or fewer stages based on their reliability needs, but the pattern is consistent.

Each stage mirrors the structure of production but typically uses fewer resources. Code promotes left to right, from development through testing, before it ever reaches production.

## Version Control & Bitbucket

**Bitbucket** is a Version Control System (VCS) built around Git, with built-in support for pull requests, branch permissions, and CI/CD pipelines. Like GitHub and GitLab, it gives teams a centralized place to collaborate on code.

Benefits of using a VCS:

1. **Centralized source of truth**: one authoritative copy of the codebase
2. **Full history**: every change is tracked, with who made it and why
3. **Controlled promotion**: code moves between environments through a defined process, not manual copying
4. **Tagging**: a tag is a named pointer to a specific commit. Teams use tags to mark releases.

## Semantic Versioning

Releases are labeled using a standard format: **MAJOR.MINOR.PATCH**

| Segment | When to increment |
|---|---|
| MAJOR | Breaking change, existing callers must update |
| MINOR | New feature, backwards compatible |
| PATCH | Bug fix, no API change |

For **internal releases**, a stage suffix is appended to communicate which environment the build targets:

- `v1.1.0-dev`: built from the dev branch, deployed to the development environment
- `v1.1.0-qa`: promoted to QA for testing
- `v1.1.0-prod`: approved for production

For **public or LTS software**, the convention is simpler. Releases are labeled `v1.0.0` or `v1.0.0-LTS` with no stage suffix, since external users do not care which internal environment it came from.

## Release Management

A `v1.1.0-qa` release is deployed to the QA staging environment for testing. If it passes, it is promoted and re-tagged as `v1.1.0-prod`. Code does not skip stages. Production releases are always promoted from QA, never deployed directly from dev.

Change Management procedures typically prohibit production updates during blackout periods and require a **CAB (Change Advisory Board)** meeting to approve change requests. This meeting gives stakeholders a chance to ask questions, understand the risk, and allocate resources to the release.

A good release management procedure includes:
- **Release strategy**: how the update is applied (blue/green, canary, recreate, etc.)
- **Downtime notification**: communicating impact to stakeholders in advance
- **Rollback plan**: how to revert if something goes wrong
- **Disaster recovery plan**: what to do if rollback also fails
- **Post-release checks**: monitoring and validation after go-live
- **Release notes**: documentation that helps the production team deploy correctly



## Hands On

In this exercise you will set up the Bitbucket repository that will serve as the foundation for the Order Match API throughout this course. You will establish a `dev` branch where all development work happens, and practice the Pull Request workflow that controls what code reaches `main`. This mirrors how real teams collaborate. No one commits directly to the production branch.

![](images/pr-workflow.png)

The diagram above shows the branch model we will use throughout this course. The `dev` branch is where all active development happens. When a change is ready, a Pull Request moves it to `main`, which always reflects production-ready code. Nothing reaches `main` without going through a PR. This controlled handoff is the foundation of a safe delivery workflow.

1. Create a Bitbucket repository named `order-match-api-<yourname>`. Use lowercase and hyphens for the name, as this follows standard naming conventions for repositories. Make sure you replace `<yourname>` with your real name.

   ![](images/bitbucket-new-repo-details.png)

2. Clone the repository into VS Code. Cloning creates a local copy of the repository on your machine, linked back to Bitbucket so you can push and pull changes as you work.

   ![](images/bitbucket-clone-repo-vscode.png)

3. Create a branch named `dev`. This is your persistent development branch, not a version tag. All work in this course begins here, not on `main`.

   ![](images/bitbucket-vscode-new-dev-branch.png)

4. Add a file named `API-Endpoints.md` to the root of the repository with the content below. This file documents the API contract for the Order Match service. In real projects, this kind of document helps developers, testers, and consumers understand what the service does before any code is written. In VS Code, when prompted to stage your changes select **Yes**, then click **Publish Branch** to push the branch to Bitbucket for the first time.

```markdown
# API Endpoints

## POST /orders
Submit a new order.
- **Body**: `{ "side": "buy" | "sell", "price": float, "quantity": int }`
- **Returns**: the created order with its assigned ID and status

## GET /orders
Returns all orders in the system.

## GET /orders/{id}
Returns a single order by ID.

## GET /queue
Returns all orders currently waiting for a match.
```

   ![](images/bitbucket-vscode-commit.png)

5. In Bitbucket, open a Pull Request from `dev` to `main`. A Pull Request is a formal request for your changes to be reviewed and accepted into the target branch. Click the button to create the PR, then fill in the details. Make sure to enable the option to delete the source branch after merge, as this keeps the repository from accumulating stale branches over time.

   ![](images/bitbucket-create-pr.png)

   ![](images/bitbucket-pr-details.png)

6. Merge the Pull Request. The changes from `dev` are now part of `main`.

   ![](images/bitbucket-merge.png)

7. Switch back to the `main` branch in VS Code and pull the latest changes. Your local copy of `main` is still at the state it was before the merge. Pulling syncs it with the remote so your local environment matches what is in Bitbucket.

   ![](images/bitbucket-main-pull.png)

> **Workflow to remember:** Whenever you start new work, switch to `main`, pull, create a new branch, make your changes, open a PR, and repeat. Never commit directly to `main`.
