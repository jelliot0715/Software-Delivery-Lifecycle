## Service Delivery & SDLC Overview

**Software Development Lifecycle (SDLC)** describes how software moves from initial requirements through development, testing, deployment, and ongoing production support.

**ITIL v4** (Information Technology Infrastructure Library) is a framework that provides a common language and best practices for managing IT services across their lifecycle.

IT systems are treated as **services**. All services managed by a team are organized within a **service portfolio**. The goal of every service is to deliver **value**.

In the early stages of the SDLC, Business Analysts (BA) identify new services or enhancements that will provide value to customers. 

BAs and Project Managers (PM) plan the delivery or enhancement of a service, considering risk, cost, ROI, and long-term support.

Product Owners (PO) manage the **service backlog** and priorities, while development and operations teams handle implementation and ongoing support. They may use SCRUM or some other project management technique.

Each role has a distinct focus within the SDLC:

| Role | Focus |
|------|-------|
| PO | “What should we build next?” |
| BA | “What problem are we solving?” |
| PM | “Are we delivering on time / within scope?” |
| Devs | “Build it.” |
| Ops/SRE | “Keep it running.” |

## Staging

To develop, test, and deploy software to production, we use separate environments called stages.
Each stage is designed to closely resemble production, while often using fewer resources to reduce cost.

Stages act as a controlled testing ground where services are validated and prepared for release.

The image below demonstrates a common staging strategy, note that different companies may structure their staging and testing differently based on their capabilities and reliability needs.

![staging](images/staging.png)

Notice how each stage mirrors the structure of production but typically uses fewer resources. The arrows show the direction of code promotion — changes flow left to right, from development through testing before ever reaching production.

In the exercise below you will simulate this staging strategy using Git branches.

## Hands On

In enterprise environments, staging strategies can vary. Some organizations use separate cloud accounts and repositories for each stage, while others use a single account and repository with logical separation.

Using a single account and repository is simpler to manage, while separating environments provides greater control and isolation.

For this exercise, you will use a single cloud account and repository.
Stages will be separated using Git branches to simulate environment promotion.


1. Create a GitHub (GH) repo for a Service named "Order Processing Service". Be sure to check the "Create Readme" option to save time. If you skipped this step, follow instructions to push a README.md containing your repo name.
    - This service handles customer orders and is critical to business operations. Any issues in production directly impact revenue.
2. The **main** branch will act as the default and production branch. Navigate to your repo **Settings->Rules->Rulesets->New Ruleset**. Select **New Branch Ruleset**. 
    - Rulesets allow you to create controls that only allow code to enter a branch if it meets requirements. Take some time to review all the options you can select.
3. Name the Ruleset "Main Protection" and set it to Active. Make sure the default branch is selected. Enable the rules below.
    - Restrict Deletion
    - Require Pull Request before Merging
    - Block force pushes
4. Save the Ruleset and update the README.md file with the content below. When you attempt to save the file, you will be prompted to create a branch, since the Ruleset will not allow direct edits to the default branch. Name the branch **feature/initial-readme**. Follow the prompts to create the PR.


```
# Order Processing Service

The Order Processing Service handles customer orders for the business. It allows users to create, view, and manage orders. This service is critical to operations, as errors may directly impact revenue and customer experience.

## Endpoints

### GET /orders
Returns a list of all orders.

- **Description**: Retrieves all existing orders in the system.
- **Use Case**: Used by support teams and dashboards to view current orders.

---

### GET /orders/{id}
Returns a specific order by ID.

- **Description**: Retrieves details for a single order.
- **Use Case**: Used to investigate or track a specific customer order.

---

### POST /orders
Creates a new order.

- **Description**: Accepts order data and creates a new order record.
- **Use Case**: Used by applications or users to submit new customer orders.

---

### PUT /orders/{id}
Updates an existing order.

- **Description**: Modifies an existing order's details.
- **Use Case**: Used to correct or update order information.

---

### DELETE /orders/{id}
Deletes an order.

- **Description**: Removes an order from the system.
- **Use Case**: Used for cleanup or handling invalid/test orders.

---

## Notes

- All changes to this service must follow the defined promotion process across environments (dev → QA → prod).
- Proper validation and testing are required before promoting changes to production.
```
