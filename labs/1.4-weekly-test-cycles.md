---

## The Weekly Test Cycle

In teams that ship regularly, testing is not ad hoc — it follows a repeatable cadence. A typical weekly cycle starts by seeding the environment with a fresh synthetic dataset, deploying the target version, running regression tests, and reviewing results. The following week the cycle begins again. This structure means the team always starts from a known, clean state and always ends with a documented record of what passed and what failed.

Each phase of the cycle maps directly to a pipeline step. Seeding is the `seed-dev` step from Lab 1.3. Deployment is `deploy-dev`. Testing is the `test-dev` step you will build in this lab. The pipeline is the test cycle.

---

## Regression Testing

A regression is when a new change breaks something that previously worked. It happens frequently in active development: a feature added to one endpoint unexpectedly alters the behavior of another, or a dependency upgrade changes how responses are serialised. Without a fixed set of checks, regressions surface in production instead of in a test environment.

Regression tests are assertions run against a deployed service to confirm it still behaves as expected. The same tests run every cycle, every week. If a test that passed last week fails this week, the pipeline makes that visible immediately — before any user is affected.

---

## Test Reporting in Bitbucket

Bitbucket Pipelines reads JUnit XML natively. pytest generates this format with a single flag: `--junitxml=<path>`. When the pipeline step declares that file as a report artifact, Bitbucket parses it and renders a **Tests** tab on the pipeline run showing pass/fail counts, individual test names, and durations. Subsequent runs build a history — over time you can see exactly when a test started failing and correlate it with the deployment that caused it.

The report is captured even if some tests fail, so you can always inspect results regardless of whether the overall pipeline step succeeded.

## Hands On

In this exercise you will write regression tests for the Order Match API, add a test pipeline step, run it against your dev environment, and then execute the complete weekly cycle end to end.

1. **Create the test files.**

   In the `tests/` directory of your forked repo (where `seed.py` already lives), add two new files.

   `tests/requirements.txt`:

   ```
   pytest==8.3.0
   requests==2.32.0
   ```

   `tests/test_api.py`:

   ```python
   import os
   import uuid
   import requests

   BASE_URL = os.environ.get("BASE_URL", "http://localhost:8000")


   def test_get_orders():
       r = requests.get(f"{BASE_URL}/orders")
       assert r.status_code == 200
       assert isinstance(r.json(), list)


   def test_post_order():
       body = {"id": str(uuid.uuid4()), "side": "buy", "price": 10, "quantity": 5, "status": "pending"}
       r = requests.post(f"{BASE_URL}/orders", json=body)
       assert r.status_code == 201
       assert "id" in r.json()


   def test_get_order_by_id():
       body = {"id": str(uuid.uuid4()), "side": "sell", "price": 12, "quantity": 3, "status": "pending"}
       r = requests.post(f"{BASE_URL}/orders", json=body)
       order_id = r.json()["id"]
       r2 = requests.get(f"{BASE_URL}/orders/{order_id}")
       assert r2.status_code == 200
       assert r2.json()["id"] == order_id


   def test_get_queue():
       r = requests.get(f"{BASE_URL}/queue")
       assert r.status_code == 200
       assert isinstance(r.json(), list)


   def test_order_not_found():
       r = requests.get(f"{BASE_URL}/orders/999999")
       assert r.status_code == 404
   ```

   Five tests cover every endpoint: listing orders, creating an order, retrieving an order by its ID, reading the match queue, and confirming that a non-existent ID returns 404. The `test_get_order_by_id` test creates an order first and then retrieves it — this is a round-trip test that validates both the write and read paths together.

2. **Run the tests locally (optional).**

   ```
   pip install -r tests/requirements.txt
   BASE_URL=http://<dev-ip>:8000 pytest tests/ -v
   ```

   Each test name is printed with PASS or FAIL, followed by a summary line showing the count and total duration. This is identical output to what the pipeline will produce.

3. **Add the test pipeline step.**

   Open `bitbucket-pipelines.yml` and add a `test-dev` custom pipeline after the existing steps:

   ```yaml
   test-dev:
     - variables:
         - name: BASE_URL
           description: "Base URL of the dev environment (e.g. http://<public-ip>:8000)"
     - step:
         name: Regression Tests – Dev
         image: python:3.11-slim
         script:
           - pip install -r tests/requirements.txt -q
           - pytest tests/ --junitxml=test-results/results.xml -v
         artifacts:
           reports:
             junit: test-results/results.xml
   ```

   `--junitxml` writes test results in the standard XML format that Bitbucket understands. The `artifacts: reports: junit:` declaration tells Bitbucket to parse that file and render the test summary. The report is written before pytest exits, so it is captured even when tests fail.

4. **Commit and push.**

   Stage `tests/requirements.txt`, `tests/test_api.py`, and the updated `bitbucket-pipelines.yml`. Push to your repo.

5. **Run the test pipeline.**

   In Bitbucket, navigate to **Pipelines → Run pipeline**. Select the `test-dev` custom pipeline. Enter the dev service's public IP as `BASE_URL` (e.g., `http://1.2.3.4:8000`). Run it.

6. **View test results.**

   When the pipeline finishes, click into the run and open the **Tests** tab. Review the test count, individual test names, pass/fail status, and duration for each test. Run the pipeline a second time — the second run appears in the history alongside the first, beginning to build the trend you would track week over week.

7. **Challenge: run the full weekly cycle.**

   Execute the complete weekly cycle against the dev environment in sequence:

   1. Run `seed-dev` (from Lab 1.3) to refresh the dataset with fresh synthetic orders
   2. Run `test-dev` to validate the environment against the new data
   3. Open the Tests tab and confirm all five tests pass

   This is the repeatable sequence a team runs at the start of each test week: seed a clean dataset, validate the environment, review results. Each run adds to the test history, giving the team a record they can refer back to when investigating a regression.
