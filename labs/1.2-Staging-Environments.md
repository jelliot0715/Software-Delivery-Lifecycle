## Staging Environments

In enterprise software delivery, code never goes directly from a developer's machine to production. It moves through a series of isolated environments, each serving a specific purpose.

Each environment runs independently. A bug introduced in dev does not affect prod. A release promoted to QA does not automatically go to prod. This isolation is what makes it safe to test and validate changes before they reach customers.

Environments commonly run different versions at the same time. Dev may be running the latest code while prod is still on the previous release. This is expected and intentional. The version running in each environment reflects how far that release has been validated, not a mistake.

---

## The Self-Service Pipeline

The repository you forked contains a Bitbucket Pipeline configured as a self-service deployment tool. Instead of triggering automatically on a branch push, it runs on demand. You select the image tag to deploy, select the target environment, and the pipeline handles the rest.

This pattern is common in teams that want control over when each environment is updated, rather than having every commit automatically propagate forward.

## Hands On

In this exercise you will connect your forked repository to AWS, provision your staging environments, and deploy different versions to dev and prod to see environment isolation in practice. You will then extend the setup to add a QA environment.

1. **Create AWS access keys and configure Bitbucket repository variables.**

   Log into the AWS Console. Click your account name in the top-right corner and copy your **Account ID**.

   Navigate to **IAM → Users → [your user] → Security credentials → Create access key**. Select **Third-party service** as the use case. Copy and save both the Access Key ID and Secret Access Key before closing the dialog. The secret cannot be retrieved again.

   In your forked Bitbucket repository, go to **Repository settings → Pipelines → Repository variables**. Add the following five variables:

   | Variable | Value |
   |---|---|
   | `AWS_ACCESS_KEY_ID` | The access key ID you just created |
   | `AWS_SECRET_ACCESS_KEY` | The secret access key you just created |
   | `AWS_DEFAULT_REGION` | `us-east-1` |
   | `AWS_ACCOUNT_ID` | Your 12-digit AWS account ID |
   | `DOCKERHUB_IMAGE` | `innovationinsoftware/order-match-api` |

2. **Provision infrastructure.**

   Exit the settings of your repository and go back to the repository home page. Navigate to **Pipelines → Run pipeline**. Select `main` branch and the `terraform-apply` custom pipeline. Run and wait for the pipeline to complete. Review the pipeline outpu. Note the resources created: the ECS cluster (`order-match-api`), IAM execution role, CloudWatch log groups, and ECS services for dev and prod.

3. **Deploy `v1.0.0` to prod.**

   Run the `deploy-prod` pipeline with `TAG` = `v1.0.0`. Wait for it to complete.

   To confirm the deployment: in the AWS Console, navigate to **ECS → Clusters → order-match-api → Services → order-match-api-prod → Tasks**. Click the running task and copy its **Public IP**. Open a browser and navigate to `http://<public-ip>:8000/docs`. You see a Swagger UI page. Use it to test your application.

4. **Deploy `v1.0.1` to dev.**

   Run the `deploy-dev` pipeline with `TAG` = `v1.0.1`. Wait for it to complete. Find the dev task's public IP the same way (under the `order-match-api-dev` service) and confirm `http://<public-ip>:8000/docs`.

   Dev and prod are now running different versions of the same service. Prod is stable on `v1.0.0`. Dev has the newer patch `v1.0.1`. This reflects a normal state during active development.

5. **Challenge: add a QA environment.**

   Study the existing files before making any changes — the code includes comments marking exactly where to add QA:
   - `task-definitions/dev.json` and `task-definitions/prod.json` — note the family names, placeholder tokens (`IMAGE_PLACEHOLDER`, `ACCOUNT_ID_PLACEHOLDER`, `REGION_PLACEHOLDER`), log group names, and resource sizing
   - The `deploy-dev` and `deploy-prod` steps in `bitbucket-pipelines.yml` — note the comment marking where to add `deploy-qa`
   - The dev and prod resources in `terraform/main.tf` — note the three comments marking where to add the QA log group, task definition, and service

   Add QA following the same pattern:
   - Create `task-definitions/qa.json` — copy `dev.json`, set family to `order-match-api-qa` and log group to `/ecs/order-match-api/qa`
   - Add a `deploy-qa` custom pipeline step to `bitbucket-pipelines.yml` referencing `order-match-api-qa` and `task-definitions/qa.json`
   - Add to `terraform/main.tf`: a log group `/ecs/order-match-api/qa`, a task definition `order-match-api-qa`, and an ECS service `order-match-api-qa`
   - Run the `terraform-apply` pipeline again
   - Run your new `deploy-qa` pipeline with `TAG` = `v1.0.1`
   - Find the QA task's public IP and confirm `http://<public-ip>:8000/docs` responds

6. **Verify all three environments.**

   Confirm each environment is running the expected version and responding:

   | Environment | Expected version | Test URL |
   |---|---|---|
   | prod | `v1.0.0` | `http://<prod-ip>:8000/orders` |
   | dev | `v1.0.1` | `http://<dev-ip>:8000/orders` |
   | QA | `v1.0.1` | `http://<qa-ip>:8000/orders` |

   To verify the version running in each environment: in the ECS Console, open the task, expand the container, and check the image tag shown under **Image**.
